# Coffee Chat with Hank @ Abel AI

## Abel AI 一句话

用 PCMCI 统计因果发现处理 20 万+ 金融变量，每日刷新因果图，每小时更新预测。

## CurioCat 一句话

用 LLM 从非结构化文本中提取因果关系，通过证据锚定和多层验证构建可交互的因果决策图。

---

## 核心差异

|      | CurioCat          | Abel AI          |
| ---- | ----------------- | ---------------- |
| 输入   | 文本（新闻/政策/假设）      | 时序数据（20万+变量）     |
| 因果发现 | LLM 语义判断          | PCMCI 统计检验       |
| 边的属性 | 机制描述 + 强度 + 证据URL | β系数 + 时滞τ + p值   |
| 统计保证 | 无（靠多层验证补偿）        | 有（p值）            |
| 适合   | 定性推演（没有历史数据的新场景）  | 定量预测（有充分观测数据的领域） |

## Talking Points

### 1. 互补性

- Abel 处理不了没有历史数据的新事件（全新政策/突发事件）
- CurioCat 处理不了需要统计严谨性的定量预测
- 融合方向：Abel 的统计因果作为 CurioCat 证据锚定的数据源？

### 2. 幻觉 vs. 显著性

- Abel 天然无幻觉（统计检验）
- CurioCat 靠 7 层验证：门控 → 偏见审计 → 证据调制 → 多样性惩罚 → 网络验证 → 溯源 → 置信区间
- 可以问：Abel 如何处理虚假因果（spurious correlation）？PCMCI 的条件独立性检验在高维场景下的表现？

### 3. Pearl 因果阶梯

- Abel 对标 Layer 2-3（干预 + 反事实），用 do-calculus
- CurioCat 的情景推演本质也是干预分析（修改边强度 → 重新传播）
- 差异：CurioCat 用 Noisy-OR 近似，Abel 用结构方程模型

### 4. 架构方向相反

- Abel：因果能力封装成 API（CAP Protocol）→ 给 LLM Agent 调用
- CurioCat：LLM 能力封装进因果流水线 → 由确定性图算法做最终计算
- 可以问：Hank 怎么看这两种架构的 tradeoff？

### 5. 工程问题

- Abel：20万变量、600万节点，怎么做到日级刷新？PCMCI 的 O(n²pτ) 如何优化？
- CurioCat：50-200 节点，瓶颈在 LLM 调用（N² 配对 → embedding 预过滤砍到 10-30%）
- 可以问：Abel 的分布式计算架构？增量更新策略？

## Hank 可能问的问题（准备好答案）

1. **LLM 做因果判断靠谱吗？**
   → 单独不靠谱，所以用多层验证：验证门控拒绝空洞机制、偏见审计检测 8 种认知偏见、证据调制让零证据边完全不传播、最终置信度由确定性图算法计算而非 LLM

2. **Noisy-OR 为什么不用贝叶斯网络？**
   → 参数少（每边一个强度值 vs. 完整 CPT）、语义直觉好、计算 O(入边数)、加 inhibition 扩展后覆盖抑制场景

3. **怎么防止 embedding 去重出错？**
   → 阈值 0.95 极高，只过滤机械重复；真正的语义判断走 LLM；宁可多留不误删

4. **规模能到多大？**
   → 当前 50-200 节点；瓶颈是 LLM 调用次数（O(n²) 配对）；embedding 预过滤（0.3 阈值）砍掉 70-90% 配对；发现层有收敛控制和每层 30 条上限
