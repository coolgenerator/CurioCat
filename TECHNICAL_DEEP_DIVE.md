## 1. 系统定位：解决什么问题

传统的 LLM 应用在做决策分析时有三个核心缺陷：

| 缺陷        | 表现                   |
| --------- | -------------------- |
| **黑箱推理**  | 给一个结论，但不知道推理路径是什么    |
| **无证据锚定** | 结论无法追溯到真实世界的数据源      |
| **不可调试**  | 改变一个假设后，无法看到下游影响如何变化 |

CurioCat 的设计目标是：**将 LLM 的语言理解能力约束在一个可审计、可调试的因果图结构中**。LLM 负责语义理解（拆解主张、判断因果），但最终的置信度计算由确定性的图算法完成——不依赖 LLM 的"感觉"。

---

## 2. 整体架构

```
用户输入（非结构化文本）
    │
    ▼
┌─────────────────────────────────────────────────┐
│  Layer 0（种子层）                                │
│  主张拆解 → 因果推断 → 偏见审计 → 证据锚定        │
└────────────────────┬────────────────────────────┘
                     │
    ┌────────────────▼────────────────┐
    │  Layer 1..N（发现层，最多 3 轮）  │◄─── 收敛条件：新主张 < 2 且新边 < 1
    │  事实发现 → 增量推断 → 审计 → 锚定 │
    └────────────────┬────────────────┘
                     │
    ┌────────────────▼────────────────┐
    │  最终阶段                        │
    │  DAG 构建 → 置信传播 → 敏感度分析 │
    └─────────────────────────────────┘
                     │
                     ▼
          可交互因果图 + 战略报告
```

**关键设计决策：分层迭代而非一次生成。**

为什么？因为原始文本通常只包含部分因果链。通过证据搜索发现新事实，再对新事实做因果推断，可以补全文本中未提及但客观存在的因果关系。每轮发现只处理"新×旧"和"新×新"的配对，避免 O(n²) 的重复计算。

## 3. 流水线详解

### 3.1 主张拆解（Claim Extraction）

**目标：** 将非结构化文本拆解为原子化、可独立评估的主张。

#### 分块策略

```
长文本 ──┬── chunk 1 (0~8000 chars) ──┐
         ├── chunk 2 (7000~15000)     ├── 逐块 LLM 提取 → 合并 → 去重
         └── chunk 3 (14000~22000) ───┘
              ↑ 1000 字符重叠区
```

- **块大小：** 8000 字符（`_MAX_CHUNK_LENGTH`）
- **重叠：** 1000 字符（`_OVERLAP_LENGTH`）
- **为什么要重叠：** 句子可能恰好被切断在两个块的边界。1000 字符的重叠保证跨边界的上下文不丢失，代价是可能产生重复提取——由下游的去重步骤处理。

#### LLM 提取

每条 claim 输出：

```json
{
  "text": "原子化主张文本",
  "type": "FACT | ASSUMPTION | PREDICTION | OPINION",
  "confidence": 0.0-1.0,
  "source_sentence": "原文逐字引用"
}
```

**四种分类的设计意图：**

| 类型         | 用途                   |
| ---------- | -------------------- |
| FACT       | 可验证，适合证据锚定           |
| ASSUMPTION | 标记为假设，提醒用户这是推理前提而非事实 |
| PREDICTION | 前瞻性判断，置信度通常较低        |
| OPINION    | 主观判断，在图中作为低权重节点      |

**为什么要记录 `source_sentence`：** 全链路审计。用户可以追溯任何一条 claim 回到原文的哪句话，验证提取是否忠实。

#### 语义去重

```python
_DEDUP_SIMILARITY_THRESHOLD = 0.95  # 余弦相似度阈值
```

1. 所有 claim 文本通过 OpenAI embedding 编码为 1536 维向量
2. 逐条比较：新 claim 与已保留 claim 的余弦相似度 > 0.95 → 标记为重复
3. 贪心策略：按提取顺序保留第一条

**为什么是 0.95 而不是更低（如 0.85）？**

这是一个**保守设计**。0.95 只过滤几乎逐字相同的表述（主要来自分块重叠），不会误删"看起来像但实际不同"的 claim。即使有两条语义相近的 claim 都保留了，它们在因果推断阶段会建立相同的因果边、获得相同的证据——冗余但不会导致错误结论。反过来，如果阈值设得太低，误删一条关键 claim 会断掉整条因果链，这是不可逆的损失。

### 3.2 因果推断（Causal Inference）

**目标：** 判断任意两条 claim 之间是否存在因果关系。

#### 两阶段过滤架构

这是整个流水线中**计算成本最高**的环节。N 条 claim 有 N×(N-1)/2 种配对，如果全部交给 LLM 判断，50 条 claim 就需要 1225 次 LLM 调用。

**解决方案：embedding 相似度预过滤 + LLM 精判。**

```
阶段 1：embedding 预过滤
┌──────────────────────────────────────┐
│ 计算 N×N 余弦相似度矩阵              │
│ 保留 similarity > 0.3 的配对         │
│ 典型过滤率：70-90%                   │
└─────────────┬────────────────────────┘
              │ 候选配对
              ▼
阶段 2：LLM 因果判断
┌──────────────────────────────────────┐
│ 每对配对独立调用 LLM                  │
│ 并发限制：5 个同时调用                │
│ 输出：方向、机制、强度、时间延迟、条件  │
└─────────────┬────────────────────────┘
              │
              ▼
阶段 3：验证门控
┌──────────────────────────────────────┐
│ 机制描述 ≥ 10 字符？                  │
│ 机制描述 ≠ 原文重述？                 │
│ 强度 ∈ [0.1, 0.95]？                 │
│ 不通过 → 丢弃该边                     │
└──────────────────────────────────────┘
```

**为什么预过滤阈值是 0.3（很低）？**

0.3 的余弦相似度只要求"话题相关"，不要求"语义相近"。这是故意宽松的——宁可多送一些配对给 LLM，也不能漏掉真正的因果关系。真正的因果判断由 LLM 完成，embedding 只负责砍掉明显不相关的配对（比如"经济政策"和"天气预报"）。

**验证门控的设计意图：**

| 门控               | 防止什么                              |
| ---------------- | --------------------------------- |
| 机制 ≥ 10 字符       | LLM 虚构一个空洞的因果关系（"它们有关系"）          |
| 机制 ≠ 原文重述        | LLM 把两条 claim 原文拼起来当作"机制"         |
| 强度 ∈ [0.1, 0.95] | LLM 输出极端值（0.0 或 1.0），通常是不确定时的默认行为 |

#### 因果边的丰富属性

每条因果边不只是"A→B 强度 0.7"，而是一个结构化描述：

```
causal_type:      direct / indirect / probabilistic / enabling / inhibiting / triggering
condition_type:   sufficient / necessary / contributing
temporal_window:  "1-3 months" / "permanent"
decay_type:       none / linear / exponential / accumulative
conditions:       ["前提条件1", "前提条件2"]
reversible:       true / false
```

**为什么需要这么多属性？**

因为下游的图算法需要它们：

- `causal_type: inhibiting` → 在置信传播中作为抑制因子
- `condition_type: necessary` → 触发 AND 逻辑门（所有前因都必须满足）
- `decay_type` → 在时间线视图中展示因果效力的衰减
- `conditions` → 在交互界面中提醒用户这条因果关系的成立前提

#### 增量推断模式

发现层不需要重新检查所有配对。如果有 50 条旧 claim 和 5 条新 claim：

- 需要检查：5×50 + 5×4/2 = 260 对（新×旧 + 新×新）
- 跳过：50×49/2 = 1225 对（旧×旧，已经检查过了）

### 3.3 偏见审计（Bias Audit）

**目标：** 检测因果推断中可能存在的认知偏见，自动降低可疑边的强度。

#### 两层审计

```
层 1：结构检查
  └─ 机制描述为空/过短？→ 强度 × 0.5

层 2：LLM 偏见检测（8 种认知偏见）
  ├─ correlation_not_causation（相关≠因果）
  ├─ survivorship_bias（幸存者偏差）
  ├─ narrative_fallacy（叙事谬误）
  ├─ anchoring_effect（锚定效应）
  ├─ reverse_causality（反向因果）
  ├─ selection_bias（选择偏差）
  ├─ ecological_fallacy（生态谬误）
  └─ confirmation_bias（确认偏差）
```

#### 惩罚机制

```
severity_penalties = {
    "low":    0.95,   # -5%
    "medium": 0.80,   # -20%
    "high":   0.60,   # -40%
}
```

**乘法叠加：** 两个"high"偏见 → 强度 × 0.6 × 0.6 = 0.36

**下限保护：** 强度 ≥ 0.05，永远不会降到 0——因为即使有偏见，这条因果关系也可能部分成立。

**为什么用乘法而不是加法？**

乘法有两个好处：(1) 自然衰减，多个偏见累积效果递减而非线性增长；(2) 不会出现负值。

### 3.4 证据锚定（Evidence Grounding）

**目标：** 用真实世界的证据为每条因果边打分，防止 LLM 编造的因果关系进入图谱。

#### 对抗式双向搜索

对每条因果边，构造两个搜索查询：

```
正向查询：  "evidence {机制描述} {原因claim前100字符}"      → 搜 3 条结果
反向查询：  "counter-evidence against {机制描述} {结果claim前100字符}" → 搜 2 条结果
```

**为什么是 3+2 而不是 5+5？**

平衡成本和效果。正面证据更重要（决定这条边是否可信），反面证据用于降低过度自信。3+2 在实测中是性价比最优的组合。

#### 四维评分体系

每条证据从四个维度加权：

```
weighted_score = relevance × credibility × freshness × tier_multiplier
```

| 维度                        | 来源     | 范围       |
| ------------------------- | ------ | -------- |
| **relevance**（相关性）        | LLM 评分 | 0.0-1.0  |
| **credibility**（可信度）      | 域名分类   | 0.2-0.95 |
| **freshness**（时效性）        | 发布日期   | 0.1-1.0  |
| **tier_multiplier**（层级权重） | 来源类型   | 0.5-1.0  |

#### 来源可信度分级

| 类型       | 可信度  | 层级     | 示例                                |
| -------- | ---- | ------ | --------------------------------- |
| academic | 0.95 | Tier 1 | arxiv.org, nature.com, .edu, .gov |
| news     | 0.75 | Tier 3 | reuters.com, bbc.com, nytimes.com |
| blog     | 0.50 | Tier 4 | medium.com, substack.com          |
| forum    | 0.35 | Tier 5 | reddit.com, stackoverflow.com     |
| social   | 0.20 | Tier 6 | twitter.com, facebook.com         |

#### 时效性衰减

| 发布时间    | 时效分数 |
| ------- | ---- |
| < 1 个月  | 1.0  |
| 1-6 个月  | 0.8  |
| 6-12 个月 | 0.6  |
| 1-3 年   | 0.4  |
| 3-5 年   | 0.25 |
| > 5 年   | 0.1  |
| 未知      | 0.5  |

#### 综合证据分数

```
support_avg = avg(正面证据加权分)
contra_avg  = avg(反面证据加权分)

raw_score = 0.5 + 0.5 × (support_avg - contra_avg)    # 范围 [0, 1]
```

**来源多样性惩罚：**

```
diversity_factor = min(1.0, 不同域名数 / (总证据数 × 0.5))
final_score = raw_score × (0.7 + 0.3 × diversity_factor)
```

如果所有证据来自同一个域名，最多降权 30%。防止"引用一堆同源文章"造成的虚假支撑。

#### 共识等级

| 正面证据占比 | 共识等级                |
| ------ | ------------------- |
| ≥ 80%  | strong_support      |
| 60-80% | moderate_support    |
| 40-60% | contested           |
| 20-40% | moderate_opposition |
| < 20%  | strong_opposition   |
| 总数 < 2 | insufficient        |

### 3.5 事实发现（Discovery）

**目标：** 从已搜集的证据片段中提取原文未提及的新事实，扩展因果图的覆盖范围。

#### 流程

```
已有因果边的证据片段
    │
    ▼
LLM 提取新主张（每批 3 条边的证据）
    │
    ▼
embedding 编码
    │
    ▼
双重去重
  ├─ vs 已有 claim（余弦 > 0.95 → 丢弃）
  └─ vs 本批次其他新 claim（余弦 > 0.95 → 丢弃）
    │
    ▼
网络验证（搜索确认新主张是否可查证）
  ├─ 有搜索结果 → 保留
  ├─ 无搜索结果 → 丢弃（可能是幻觉）
  └─ 搜索出错 → 保守保留
```

**为什么需要网络验证？**

防止循环推断。LLM 可能从证据片段中"发现"一个实际上不存在的新事实。网络验证确保新主张至少能在公开信息中找到印证。

**收敛控制：**

- 每层最多 30 条新 claim（`_MAX_CLAIMS_PER_LAYER`）
- 新主张 < 2 条且新边 < 1 条时停止
- 最多 3 轮发现层

### 3.6 DAG 构建

**目标：** 将因果边集合转化为有向无环图（DAG）。

#### 三步处理

```
步骤 1：弱边剪枝
  └─ strength < 0.1 的边直接移除

步骤 2：环路断裂
  ┌─ 尝试拓扑排序
  ├─ 失败 → 找到一个环
  ├─ 计算环中每条边的 score = strength × evidence_score
  ├─ 移除 score 最低的边
  └─ 重复直到无环

步骤 3：逻辑门判定
  └─ 对每个有 ≥2 条入边的节点：
     ├─ 所有入边 causal_type == "enabling" → AND 门
     ├─ 任何入边 condition_type == "necessary" → AND 门
     └─ 其他 → OR 门（默认）
```

**为什么断裂最弱边？**

因果图中的环通常意味着反馈循环（A→B→C→A），但 DAG 不允许环。断裂最弱边是信息损失最小的策略——保留了因果链中最有证据支撑的部分。

**AND/OR 门的意义：**

- **OR 门：** 多个独立原因中任何一个都可以导致结果（Noisy-OR 模型）
- **AND 门：** 所有前因必须同时成立，结果才会发生

这直接影响下游的置信传播算法。

---

### 3.7 置信传播（Belief Propagation）

**目标：** 从根节点（无前因的 claim）出发，沿拓扑顺序向下传播置信度。

#### 核心算法：Noisy-OR with Inhibition

**证据调制函数：**

```
modulation = evidence_score ^ 1.5
```

| evidence_score | modulation | 含义          |
| -------------- | ---------- | ----------- |
| 0.0            | 0.0        | 无证据 → 完全无效  |
| 0.5            | 0.35       | 部分证据 → 保守传播 |
| 1.0            | 1.0        | 充分证据 → 完全传播 |

**为什么用 1.5 次方而不是线性？**

1.5 次方对弱证据的惩罚更重。线性映射下，0.5 的证据还能传播 50% 的强度；1.5 次方下只传播 35%。这是一个**保守设计**——宁可低估也不高估。

**根节点：**

```
belief = confidence  （LLM 给出的初始置信度）
```

**OR 门节点（默认）：**

```
effective_strength = strength × modulation(evidence_score)

# 正向因果（direct, enabling, etc.）
noisy_or_product = ∏(1 - parent_belief × effective_strength)

# 抑制因果（inhibiting）
inhibition_factor = ∏(1 - parent_belief × effective_strength)

belief = (1 - noisy_or_product) × inhibition_factor
```

**AND 门节点：**

```
belief = ∏(parent_belief × effective_strength)
```

**为什么选 Noisy-OR 而不是贝叶斯网络？**

1. Noisy-OR 参数少——每条边只需要一个强度值，不需要完整的条件概率表
2. 语义直觉好——"多个独立原因都可以触发结果"符合大多数因果场景
3. 加上 inhibition 扩展后，可以建模"A 抑制 B"的场景
4. 计算复杂度 O(入边数)，贝叶斯网络的精确推断是 NP-hard

#### 置信区间

通过扰动分析计算不确定性：

```
对每条边施加 ±0.1 的强度扰动
重新传播 → 得到每个节点的 belief_low 和 belief_high
```

## 4. 图算法层

置信传播之上，还有四个分析算法：

### 4.1 关键路径（Critical Path）

**算法：** 拓扑排序上的动态规划，求最长加权路径。

```
edge_weight = strength × modulation(evidence_score)

对每个节点按拓扑序：
  dist[node] = max(dist[parent] + edge_weight)  对所有非抑制入边
```

回溯得到从根到叶的最强因果链。

**用途：** 告诉用户"这个结论最依赖哪条推理链"。

### 4.2 敏感度分析（Sensitivity Analysis）

**算法：** 逐边扰动 ±20%，测量下游节点的最大置信度变化。

```
对每条边 (u, v)：
  扰动 strength + 0.20 → 重新传播 → 测量所有 v 的后代节点的 belief 变化
  扰动 strength - 0.20 → 重新传播 → 测量所有 v 的后代节点的 belief 变化
  sensitivity = max(两次变化的最大值)
```

**用途：** 找到"改变哪条因果边对全局影响最大"，指导用户优先验证高敏感度的假设。

### 4.3 路径查找（Path Finder）

**算法：** 枚举从根节点到目标节点的所有简单路径，按复合概率排序。

```
compound_probability = ∏(strength × modulation(evidence_score))  路径上所有边
```

**限制条件（防止组合爆炸）：**

- 最多返回 10 条路径
- 路径长度 ≤ 6
- 最多探索 200 个候选
- 硬时间限制 3 秒

**用途：** 回答"为什么系统认为这条 claim 可信？"——展示所有支撑它的推理路径。

### 4.4 焦点子图（Focus Subgraph）

**算法：** 从焦点节点做双向 BFS，默认 2 跳。

**用途：** 在稠密图中只展示用户关心的局部因果网络。

## 5. 反幻觉设计

CurioCat 的核心挑战是：**用一个会幻觉的 LLM 来构建可信的因果图**。解决方案是多层验证，每一层针对一种幻觉路径：

| 幻觉路径       | 防御层   | 机制                                    |
| ---------- | ----- | ------------------------------------- |
| LLM 编造因果关系 | 验证门控  | 机制描述必须 ≥10 字符且不能是原文重述                 |
| LLM 给极端置信度 | 强度钳位  | 因果强度被钳制在 [0.1, 0.95]                  |
| 相关当因果      | 偏见审计  | 检测 8 种认知偏见，自动降低强度                     |
| 无证据支撑的主张   | 证据调制  | evidence_score=0 → modulation=0，完全不传播 |
| 证据来源单一     | 多样性惩罚 | 单一域名最多降权 30%                          |
| 循环推断       | 网络验证  | 发现的新主张必须通过网络搜索验证                      |
| 累积偏差       | 置信区间  | 扰动分析展示每个节点对输入变化的敏感程度                  |

**设计哲学：LLM 做语义理解，确定性算法做数值计算。**

LLM 负责的事情：拆解 claim、判断因果方向和机制、评分证据相关性、检测偏见。

LLM **不**负责的事情：计算最终置信度、传播信念、找关键路径、做敏感度分析——这些全部由 NetworkX 图算法和 NumPy 数值计算完成。

## 6. 模型选型与参数策略

### LLM 配置

| 任务     | 默认模型   | Temperature | 为什么              |
| ------ | ------ | ----------- | ---------------- |
| 主张拆解   | GPT-4o | 0.3         | 需要创造性理解但不能太发散    |
| 因果推断   | GPT-4o | 0.2         | 更高确定性，因果判断不应有随机性 |
| 偏见检测   | GPT-4o | 0.3         | 需要一定灵活度发现隐含偏见    |
| 证据相关性  | GPT-4o | 0.3         | 平衡精度和召回          |
| 事实发现   | GPT-4o | 0.3         | 需要从证据中发现新信息      |
| 语义等价判断 | GPT-4o | 0.2         | 二值判断，需要高确定性      |
| 情景分析   | GPT-4o | 0.3         | 需要综合分析能力         |
| 战略顾问   | GPT-4o | 0.3         | 需要创造性建议          |

**为什么默认 GPT-4o？**

- 需要强推理能力（因果判断、偏见检测）
- 需要可靠的结构化输出（JSON schema strict mode）
- 支持 Anthropic Claude 作为备选（配置切换即可）

**Temperature 策略：** 判断类任务（因果推断、语义等价）用 0.2，分析类任务用 0.3。所有任务都低于 0.5——这不是创意写作，是需要可复现的分析。

### Embedding 配置

| 参数   | 值                      | 为什么                  |
| ---- | ---------------------- | -------------------- |
| 模型   | text-embedding-3-small | 性价比最优，1536 维足够做相似度过滤 |
| 维度   | 1536                   | 模型默认维度               |
| 批量大小 | 100                    | OpenAI API 单次上限      |

**注意：** 即使 LLM 切换到 Anthropic，embedding 始终使用 OpenAI——因为 Anthropic 不提供 embedding API。

### 结构化输出策略

| 提供商       | 方法                                              | 可靠性                     |
| --------- | ----------------------------------------------- | ----------------------- |
| OpenAI    | `response_format: json_schema` + `strict: true` | API 层面强制，100% 符合 schema |
| Anthropic | JSON prefill（助手消息以 `{` 开头）+ 手动解析                | 依赖模型遵循指令，需要异常处理         |

## 7. 与现有方案的差异

### vs. GraphRAG（微软）

| 维度       | CurioCat              | GraphRAG         |
| -------- | --------------------- | ---------------- |
| **图的含义** | 因果图（有向，带强度和置信度）       | 知识图谱（实体-关系三元组）   |
| **目标**   | 辅助决策（看到推理过程）          | 增强检索（回答问题更准确）    |
| **边的语义** | 因果关系（A 导致 B，强度 0.7）   | 语义关系（A 是 B 的一部分） |
| **置信度**  | 有——Noisy-OR 传播 + 证据调制 | 无——没有置信度计算       |
| **证据锚定** | 有——每条边都有网络证据支撑        | 无——依赖原始文档        |
| **可调试性** | 高——可以修改任何边的强度并看到下游影响  | 低——图是静态的         |
| **偏见检测** | 有——8 种认知偏见            | 无                |

### vs. 传统贝叶斯网络

| 维度       | CurioCat            | 贝叶斯网络                  |
| -------- | ------------------- | ---------------------- |
| **图的构建** | LLM 自动从文本提取         | 领域专家手工搭建               |
| **参数获取** | LLM 估计强度 + 证据校准     | 统计数据或专家标注              |
| **推断算法** | Noisy-OR（简洁，每边一个参数） | 精确推断（需要完整 CPT，NP-hard） |
| **可扩展性** | 处理 50-200 节点的因果图    | 大规模图的精确推断不可行           |
| **证据集成** | 实时网络搜索              | 依赖预先收集的数据集             |

### vs. LLM 直接回答

| 维度       | CurioCat         | ChatGPT 直接回答 |
| -------- | ---------------- | ------------ |
| **透明度**  | 每步推理都可审查         | 黑箱           |
| **可追溯性** | claim→原文，边→证据URL | 无            |
| **可调试性** | 修改假设 → 看下游变化     | 重新问一遍，结果可能不同 |
| **幻觉控制** | 7 层验证            | 无            |
| **一致性**  | 确定性图算法，相同输入相同结果  | 每次回答可能不同     |

### vs. 因果发现算法（PC/FCI/GES）

| 维度       | CurioCat          | 传统因果发现            |
| -------- | ----------------- | ----------------- |
| **输入**   | 非结构化文本            | 结构化数据表（观测数据）      |
| **发现方法** | LLM 语义理解          | 条件独立性检验           |
| **适用场景** | 政策分析、新闻事件、战略假设    | 有大量观测数据的领域        |
| **统计保证** | 无——依赖 LLM 判断和证据搜索 | 有——在满足假设条件下有理论保证  |
| **可解释性** | 高——每条边有机制描述和证据    | 中——知道有因果但不一定知道为什么 |
